LLMListen — Build Plan

Capture system audio (calls, meetings, videos) for up to 2 hours · transcribe locally · hand the text to any LLM. Plan as of June 10, 2026.

Verdict: don't build the hard part — wrap it.
Your Mac runs macOS 26.5, which ships Apple's new on-device transcription engine (SpeechAnalyzer) and native system-audio taps. The open-source CLI yap (CC0 / public domain, single binary via Homebrew) already combines both: yap listen transcribes system audio live, yap listen-and-dictate adds your microphone — exactly what a call needs. The plan: a thin wrapper script llmlisten around yap that adds session management, transcript files, and a one-flag LLM handoff. No accounts, no cloud, no virtual audio drivers, no background daemons.

3Architecture

CaptureCore Audio tap: system audio (+ optional mic) — handled by yap
TranscribeSpeechAnalyzer, live & on-device, locale de-DE or en-US
StoreTimestamped Markdown in ~/LLMListen/transcripts/ + clipboard
Hand offPipe to claude -p, or any configured LLM command / API
Target usage — the whole tool is one command: 
# start before the call; Ctrl-C (or auto-stop at 2 h) ends it
llmlisten                          # system audio only (videos, recordings)
llmlisten --mic                    # call mode: both sides of the conversation
llmlisten --lang de                # German session

# after the call
llmlisten last                     # print / copy the latest transcript
llmlisten last --to claude "summarize, list action items"
llmlisten last --to "uvx llm -m gpt-5"   # any LLM you like

4Build phases

Phase 0 — Validate the foundation ~20 min
  • brew install yap, grant the one-time Screen Recording permission (that's how macOS gates system-audio capture).
  • Smoke test: play a YouTube video, run yap listen, check quality in English and German.
  • Endurance test: run against a long podcast for 45+ min to confirm stability before trusting it with a 2 h call.
Phase 1 — The llmlisten wrapper ~1 h
  • Single shell script (~100 lines), installed on your PATH. No build step, trivially editable.
  • Starts yap listen / listen-and-dictate, streams text live to the terminal and to ~/LLMListen/transcripts/2026-06-10_1430.md with a date/time header.
  • Auto-stop safety at 2 h; auto-restart + append if yap ever crashes mid-session (so a hiccup at minute 90 doesn't lose the call).
  • On stop: copies transcript to clipboard, prints word count + file path.
Phase 2 — LLM handoff ~30 min
  • llmlisten last --to <cmd> "prompt" pipes the transcript into any command: claude -p by default, but any CLI/API wrapper works — that's your "LLM of choice".
  • Tiny config file (~/.llmlistenrc) for defaults: language, mic on/off, favorite LLM command.
Phase 3 — Menu bar app chosen per Emil's request ~1 h
  • Instead of a CLI wrapper: a tiny native menu bar app (one Swift file, AppKit only) — waveform icon in the top bar, click to start/stop, red dot + elapsed timer while recording, "Summarize with Claude" in the menu.
  • Phases 1–3 merge into this single app; same behavior (transcript files, 2 h cap, crash restart, clipboard, LLM handoff).
  • Future option: --keep-audio safety net that also saves raw audio for important calls.

5Risks & honest caveats

No raw audio by default. yap transcribes live and discards audio. If transcription stumbles (heavy accents, bad call quality), there's no recording to retry from. Mitigation: the Phase 3 --keep-audio flag for high-stakes calls.
2-hour stability is assumed, not proven. No documented duration limit, but the endurance test in Phase 0 is mandatory before relying on it. The auto-restart logic in Phase 1 is the belt-and-suspenders.
One language per session. SpeechAnalyzer transcribes in one locale at a time. Mixed German/English calls will favor whichever language you set; fully mixed conversations are the weak spot of every on-device engine.
Consent. Recording calls requires the other party's consent in Germany (§201 StGB) and many other jurisdictions. The tool will print a brief reminder on start.

6Plan B — from scratch only if yap falls short

If Phase 0 reveals dealbreakers (instability, poor quality), build a small Swift CLI (~400 lines, 1–2 days) using two proven open-source references: AudioCap by Guilherme Rambo (BSD-2, the canonical Core Audio process-tap implementation) for capture, plus Apple's SpeechAnalyzer API for transcription — or whisper.cpp if Apple's engine itself is the problem. Same wrapper interface on top, so nothing else changes.
